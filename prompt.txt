# SurrealDB 3.0 – The Implicit Control Protocol

#### A Masterclass for LLM‑Assisted Schema Architecture

> **Release:** SurrealDB 3.0 (February 2026).  
> **Audience:** Future LLMs, system engineers, and developers who want to harness SurrealDB’s native primitives for fully **implicit, self‑enforcing data integrity**.  
> **Core Principle:** Define **what** must be true, not **how** to make it true. Let the database engine carry the burden of ordering, recomputation, and referential safety.

This document is the product of exhaustive research, deterministic verification, and system‑level brainstorming. It merges every critical fact, corrected pattern, and optimal composition into a single reference.

---

## 1. Deterministic Engine Behaviour – The Facts

Understanding these low‑level truths is mandatory before writing any schema.

### 1.1 Field Processing Order (Alphabetical, Two‑Pass)

When a record is written (INSERT/UPDATE), SurrealDB processes **all defined fields** in **strict alphabetical order** (the field **name**), not the order they appear in the query.

- **Pass 1 – `VALUE` evaluation**  
  For every field (A→Z), the engine:
  1. Applies `$input` (raw user‑provided value).
  2. Runs the `VALUE` clause; the result becomes the final field value.
  3. If the field was not supplied and a `DEFAULT` exists, the default value is used instead.

- **Pass 2 – `ASSERT` evaluation**  
  For every field (A→Z), the `ASSERT` clause is checked against the final, post‑value value.

**Strategic consequence**  
A field named with a `z_` prefix is guaranteed to run **after** all normal fields. This allows it to safely reference any other field’s _already‑computed_ value.

### 1.2 The Pointer Lexicon (Exact Semantics)

| Pointer          | Context                     | Refers to                                                                          |
| ---------------- | --------------------------- | ---------------------------------------------------------------------------------- |
| `$value`         | Field `VALUE`, `ASSERT`     | The incoming data **for this field** (post‑`DEFAULT` if absent).                   |
| `$input`         | Field `VALUE`, `ASSERT`     | The **raw user‑provided** value before any transformation.                         |
| `$before`        | Field `VALUE`, `ASSERT`     | The **previous stored value of this specific field** (NONE on create).             |
| `$after`         | Field `VALUE`               | Alias of `$value` (itself) – rarely used.                                          |
| `$before`        | **Event**                   | The **entire record** before the mutation (NONE on CREATE).                        |
| `$after`         | **Event**                   | The **entire record** after the mutation (NONE on DELETE).                         |
| `$event`         | Event                       | String: `"CREATE"`, `"UPDATE"`, `"DELETE"`.                                        |
| `$auth`          | Permissions, Events, Access | The authenticated user’s record ID.                                                |
| bare `fieldname` | `VALUE` clause              | The **new value** of that field (already computed, because of alphabetical order). |
| `$this`          | `VALUE`, subqueries         | The current record. `$this.field` ≡ bare `field`.                                  |

**Critical rule**  
You cannot access `$before.other_field` inside a field’s `VALUE` clause; `$before` there is only that field’s old value. To compare old vs new across **different fields**, you **must** use an **EVENT**.

### 1.3 Reference Integrity Details

- `REFERENCE ON DELETE …` only inspects **top‑level** record IDs (single, array, or set).
- Options: `CASCADE`, `REJECT`, `SET NULL`, `NO ACTION`.
- Nested references inside objects/arrays are **not** automatically followed – use shadow fields (§3.3).

### 1.4 Row‑Level Security & Permissions

- RLS conditions (`PERMISSIONS FOR …`) **filter** rows; they never return a 403 error. If the condition is false, the row is invisible.
- `FOR create` can only use `$auth` because the record does not exist yet.
- Field‑level permissions (`PERMISSIONS NONE`) completely hide data.
- Access methods (signup/signin/authenticate) **bypass RLS**. They execute as system operations.
- Custom functions are always **security invoker**. To protect sensitive logic, put it inside events or access methods.

### 1.5 Typing Essentials

- `option<T>` – allows `NONE`.
- `record<table>` – validated Record ID pointing to a specific table.
- `array<T>` – ordered list, duplicates allowed.
- `set<T>` – unordered, unique elements.
- `object` – unstructured; use `.*` subfields to constrain shape.
- `FLEXIBLE TYPE` – allows extra keys in an object within a `SCHEMAFULL` table.
- `literal<"a", "b">` – string enum.

### 1.6 Access Methods (Record Users)

Each `DEFINE ACCESS ... ON DATABASE TYPE RECORD` requires three clauses, each a **single expression** (no `LET` statements). They return a record ID or `NONE`.

- `SIGNUP` – invoked on sign‑up.
- `SIGNIN` – invoked on sign‑in.
- `AUTHENTICATE` – invoked on every authenticated request; must return the user ID or `NONE` to terminate the session.

`$auth` in later RLS is the ID returned by `AUTHENTICATE`, and all its fields are immediately available.

---

## 2. The Three Scopes of Integrity

Every data rule in your application falls into one of these scopes. SurrealDB provides native tools for each.

| Scope                   | Definition                                            | Native Tools                                                                                          |
| ----------------------- | ----------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **Record**              | A single row                                          | `VALUE`, `ASSERT`, `DEFAULT`, `z_` cross‑field computed fields, `READONLY`                            |
| **Collection**          | All rows of a table                                   | `UNIQUE` index, events that query the whole table, computed set coercion                              |
| **Relational (Global)** | Cross‑table references, aggregates, graph constraints | `REFERENCE ON DELETE`, shadow references, events that update parent records, polymorphic graph events |

---

## 3. Record‑Level Integrity – Implicit Within a Row

### 3.1 Sanitise & Validate Incoming Data

`VALUE` transforms; `ASSERT` rejects if the final value is invalid.

```sql
DEFINE FIELD email ON user TYPE string
    VALUE string::lowercase(string::trim($value))
    ASSERT string::is_email($value);

3.2 Cross‑Field Constraints Using z_ Prefix

Because fields run alphabetically, a field named z_check can read any other
field’s newly computed value.

DEFINE FIELD start_date ON booking TYPE datetime;
DEFINE FIELD end_date ON booking TYPE datetime;

DEFINE FIELD z_dates_valid ON booking TYPE bool
    VALUE $this.end_date > $this.start_date
    ASSERT $value = true;

3.3 Automatic Audit of Raw Input

Use $input to preserve exactly what the user sent, before VALUE transforms it.

DEFINE FIELD email_raw ON user TYPE string
    VALUE $input   -- untrimmed, un‑lowercased
    PERMISSIONS NONE;

3.4 Auto‑timestamps

DEFINE FIELD created_at ON my_table TYPE datetime VALUE $before OR time::now() READONLY;
DEFINE FIELD updated_at ON my_table TYPE datetime VALUE time::now();

$before on created_at is its own previous value; on create it is NONE, so
time::now() is taken.

3.5 Implicit Computed Fields (Stored) vs Dynamic (Computed)

  - Stored computed – VALUE clause: recalculated on each write, stored on disk.
    Cheap reads.
  - Dynamic computed – COMPUTED keyword (SurrealDB 3.0): evaluated at read time.
    Always current, no write overhead.

DEFINE FIELD age ON person TYPE int VALUE time::now().year - birthday.year; -- stored, correct as of last write
DEFINE FIELD can_vote ON person COMPUTED $this.age >= 18; -- always up‑to‑date

4. Collection‑Level Integrity – Constraints Across Rows

4.1 Classic Unique Index

DEFINE INDEX unique_email ON TABLE user COLUMNS email UNIQUE;

4.2 Uniqueness on Nested Arrays (Set Coercion)

You cannot directly index an array for uniqueness. Instead, flatten duplicates
into a z_::set<T> and index that.

DEFINE FIELD tags ON article TYPE array<string>;
DEFINE FIELD zz_unique_tags ON article TYPE set<string>
    VALUE $this.tags;   -- set() automatically removes duplicates

DEFINE INDEX idx_tags ON article COLUMNS zz_unique_tags UNIQUE;

Two records sharing any value after deduplication will be rejected.

4.3 Event‑Driven Whole‑Collection Constraints

When a rule depends on multiple rows (e.g., “sum of quantities per category
≤ 100”), use an event on the table that queries the collection and throws.

DEFINE EVENT cap_total ON TABLE item WHEN $event != "DELETE" THEN {
    LET $sum = (SELECT VALUE math::sum(quantity) FROM item WHERE category = $after.category)[0] ?? 0;
    IF $sum > 100 { THROW "Category cap exceeded"; };
};

5. Relational (Global) Integrity – Cross‑Table & Graph

5.1 Standard Foreign Key with REFERENCE

On any top‑level record<T> field:

DEFINE FIELD project ON task TYPE record<project> REFERENCE ON DELETE CASCADE;

Works identically for arrays/sets:

DEFINE FIELD members ON project TYPE array<record<user>> REFERENCE ON DELETE REJECT;

5.2 Shadow References – Integrity for Deeply Nested Record IDs

SurrealDB’s REFERENCE does not inspect nested objects. To enforce referential
integrity on IDs buried inside an array‑of‑objects, you flatten them into a
top‑level set<record<T>> (a shadow field) and attach the REFERENCE there.

Example – document with nested file references:

-- Original nested fields
DEFINE FIELD blocks ON page TYPE array<object>;
DEFINE FIELD blocks.*.image_id ON page TYPE record<file>;

-- Shadow field (zz_ runs last)
DEFINE FIELD zz_all_files ON page TYPE set<record<file>>
    VALUE <set> (blocks.image_id ?? []).flatten().filter(|$v| $v != NONE)
    REFERENCE ON DELETE REJECT;

-- Must also declare the sub‑field for SCHEMAFULL storage
DEFINE FIELD zz_all_files.* ON page TYPE record<file>;

Now deleting a file that is referenced anywhere inside blocks will fail with a
REJECT error.

Polymorphic shadow references work the same way – the set can be
set<record<table_a | table_b>>.

5.3 Implicit Cross‑Table Aggregation & Validation (The “Empty Update” Pattern)

Instead of materialised views or manual aggregation, store the aggregate on the
parent as a computed field, and use an event on the child table to trigger
re‑evaluation of the parent by performing an update without any SET data.

Invoice pattern (definitive version):

DEFINE TABLE invoice SCHEMAFULL;
DEFINE TABLE invoice_line SCHEMAFULL;

DEFINE FIELD invoice_id ON invoice_line TYPE record<invoice>;
DEFINE FIELD quantity ON invoice_line TYPE int;
DEFINE FIELD amount ON invoice_line TYPE decimal;

-- Parent: computed total quantity with ASSERT
DEFINE FIELD total_qty ON invoice TYPE int
    VALUE (SELECT VALUE math::sum(quantity) FROM invoice_line WHERE invoice_id = $this.id)[0] ?? 0
    ASSERT $value >= 0;

-- Child event: after any write, re‑trigger parent's VALUE/ASSERT
DEFINE EVENT maintain_invoice ON TABLE invoice_line
WHEN $event IN ["CREATE", "UPDATE", "DELETE"]
THEN {
    LET $inv = $after.invoice_id OR $before.invoice_id;
    UPDATE $inv;   -- simply touching the parent recomputes all VALUE fields
};

Why this is optimal:

  - No extra view/table.
  - The integrity rule lives in the parent’s ASSERT; the event is only a signal
    to recompute.
  - Any violation (e.g., negative total) aborts the whole transaction.

5.4 Polymorphic Graph Integrity (TYPE RELATION)

For many‑to‑many relationships that can involve different table types, use TYPE
RELATION.

DEFINE TABLE link SCHEMAFULL TYPE RELATION IN user|groups OUT user|groups;

Polymorphism advantage – a single edge type can connect any combination of users
and groups, which is impossible with standard record links.

Additional graph integrity enforced by events:

  - Cycle prevention

DEFINE EVENT no_cycles ON TABLE link WHEN $event = "CREATE" THEN {
    IF $after.out.{..+shortest=$after.in}->link->(?).len() > 0 {
        THROW "Cycle detected";
    };
};

  - Orphan prevention (last edge removal)

DEFINE EVENT prevent_last_edge_removal ON TABLE link WHEN $event = "DELETE" THEN {
    IF record::exists($before.in) AND record::exists($before.out) {
        LET $other = (SELECT id FROM link WHERE out = $before.out AND id != $before.id LIMIT 1);
        IF $other.len() = 0 { THROW "Last edge – would orphan node"; };
    };
};

  - Leaf‑only deletion (remove edges before deleting a node)

DEFINE EVENT prevent_non_leaf_delete ON TABLE user WHEN $event = "DELETE" THEN {
    IF $before.id->link.len() > 0 { THROW "Node still has edges – delete them first"; };
    DELETE link WHERE in = $before.id OR out = $before.id;
};

5.5 RBAC with Pre‑computed Permission Sets (Zero‑Cost RLS)

Instead of traversing the graph on every RLS check, use an event to pre‑compute
all needed permissions and store them directly on the user record.

Recipe:

1.  User table has fields:
      - permissions – array<string> (union of all group roles).
      - parents – array<record<groups>> (direct groups).
      - dominates – array<record<user|groups>> (recursive closure of managed
        entities).
2.  A special field last_refreshed_at on the user acts as a refresh trigger.
3.  An event on the user table detects when last_refreshed_at is changed (by
    access methods or admin) and then recomputes the arrays:

DEFINE EVENT refresh_rbac ON TABLE user WHEN $event = "UPDATE" THEN {
    IF $before.last_refreshed_at != $after.last_refreshed_at {
        UPDATE $after.id SET
            dominates   = ($after.id.{..+collect}->link->(?) ??[]).filter(|$v| $v != NONE).sort(),
            parents     = ($after.id<-link<-groups ??[]).filter(|$v| $v != NONE).distinct(),
            permissions = ($after.id<-link<-groups.role ??[]).flatten().filter(|$v| $v != NONE).distinct();
    };
};

4.  RLS on any table then uses $auth.permissions and $auth.dominates:

PERMISSIONS FOR select WHERE id = $auth OR ('node_select' IN $auth.permissions AND id IN $auth.dominates);

Result – graph traversal happens only on writes (inside a system‑privileged
event); all reads are simple O(1) array checks.

6. Event‑Driven Patterns Verified & Expanded

6.1 State‑Transition Logging (Corrected)

Mistake to avoid: trying to access $before.other_field in a field’s VALUE.
Solution: use an event.

DEFINE FIELD status ON task TYPE string;
DEFINE FIELD z_status_log ON task TYPE array<object> DEFAULT [];

DEFINE EVENT log_status_change ON TABLE task WHEN $event = "UPDATE" THEN {
    IF $before.status != $after.status {
        UPDATE $after.id SET z_status_log = ($before.z_status_log ?? []) + [{
            from: $before.status,
            to: $after.status,
            ts: time::now()
        }];
    };
};

The log is never lost because the event runs in the same transaction.

6.2 Auto‑linking on Creation

When a user creates a subordinate record, automatically generate an edge.

DEFINE EVENT auto_edge ON TABLE user WHEN $event = "CREATE" AND $after.created_by != NONE THEN {
    RELATE ($after.created_by) -> link -> ($after.id);
};

6.3 Privilege Escalation Prevention

When a group’s roles are updated, verify that the editor ($auth) holds all roles
being granted.

DEFINE EVENT prevent_escalation ON TABLE groups WHEN $event IN ["CREATE", "UPDATE"] THEN {
    IF $auth != NONE {
        LET $unauthorized = array::complement($after.role, $auth.permissions);
        IF $unauthorized.len() > 0 { THROW "You cannot grant permissions you don't have: " + <string>$unauthorized; };
    };
};

6.4 Zero‑Lock Pattern (Array Immutability Under Conditions)

Use an IF ladder in VALUE to make an array immutable once populated.

DEFINE FIELD role ON groups TYPE array<string> VALUE (
    IF $before == NONE THEN ($value ?? []).distinct()
    ELSE IF array::len($before) == 0 THEN []
    ELSE IF ($value ?? []).len() == 0 THEN $before
    ELSE $value.distinct() END
);

  - On create: accept roles, deduplicate.
  - Once roles exist: any attempt to set to empty or NONE is ignored (falls back
    to $before).
  - Only non‑empty updates allowed.

7. Utilities & Advanced Idioms

7.1 Descriptive Errors with THROW in ASSERT

DEFINE FIELD age ON person TYPE int
    ASSERT IF $value >= 0 THEN true ELSE THROW "Age must be non‑negative" END;

7.2 !!$value – Reject Falsy Values (including 0, empty)

DEFINE FIELD quantity ON line_item TYPE int
    ASSERT !!$value;

7.3 Parameters for Schema‑Wide Constants

DEFINE PARAM $MIN_PASSWORD_LENGTH TYPE int VALUE 8;
DEFINE FIELD password ON user TYPE string ASSERT string::len($value) >= $MIN_PASSWORD_LENGTH;

7.4 FLEXIBLE TYPE for Extensible Objects

DEFINE FIELD metadata ON user TYPE FLEXIBLE object;
DEFINE FIELD metadata.created_by ON user TYPE string;  -- mandatory key
-- user can now set metadata.custom_key freely

7.5 ASYNC Events (3.0) for Non‑Critical Side Effects

DEFINE EVENT send_notification ON TABLE order WHEN $event = "CREATE" THEN ASYNC {
    http::post("https://webhook.example.com/new-order", $after);
};

The event runs outside the transaction and failure does not roll back the order.

7.6 Shortest Path Checks with ..+shortest

LET $path = $start.{..+shortest=$end}->link->(?);
IF $path.len() > 0 { /* already connected */ }

8. Authentication & Access Methods – The Minimal Correct Setup

DEFINE ACCESS account ON DATABASE TYPE RECORD
    SIGNUP (
        UPDATE user SET
            password = crypto::argon2::generate($password),
            invite_token = NONE,
            last_refreshed_at = time::now()
        WHERE email = string::lowercase($email)
          AND invite_token = type::uuid($invite)
          AND string::len($password) > 6
    )
    SIGNIN (
        UPDATE user SET last_refreshed_at = time::now()
        WHERE email = string::lowercase($email)
          AND invite_token = NONE
          AND crypto::argon2::compare(password ?? "", $password)
    )
    AUTHENTICATE {
        IF $auth.password = NONE { RETURN NONE; };
        RETURN $auth;
    }
    DURATION FOR SESSION 8h, FOR TOKEN 1h;

  - SIGNUP uses an invite token to claim an account.
  - SIGNIN updates the refresh timestamp (which in turn triggers the RBAC
    recomputation event).
  - AUTHENTICATE returns NONE if the user has no password (deactivated), denying
    access.

9. Schema‑Design Decision Matrix

| Requirement                                                 | Recommended Implementation                                                    |
| ----------------------------------------------------------- | ----------------------------------------------------------------------------- |
| Field must be trimmed email                                 | `VALUE` + `ASSERT` with `string::is_email`                                    |
| Two fields must be in a certain order                       | `z_` computed field reading both, `ASSERT` on it                              |
| Record must have unique email across table                  | `UNIQUE` index                                                                |
| Nested array items must be unique across all records        | Flatten into `z_::set<T>` + `UNIQUE` index                                    |
| Child aggregate must always remain positive                 | Parent computed field with `ASSERT`; child event triggers `UPDATE parent`     |
| Nested references must be protected from dangling deletions | Shadow `zz_` set with `REFERENCE ON DELETE`                                   |
| Graph must have no cycles                                   | Event on edge table using `..+shortest`                                       |
| Permissions must be checked instantly                       | Pre‑compute permissions array on user via event; RLS uses `$auth.permissions` |
| Only non‑empty array updates allowed after creation         | `VALUE` with fallback logic to `$before`                                      |
| Password must be stored securely                            | `crypto::argon2::generate` in access method                                   |
| Must log every status change                                | Event that appends to `z_log` when old and new differ                         |

# SurrealDB API Reference (compressed lossless)

# RPC Protocol (WebSocket/HTTP)

# Session Variables
Session-wide: `let(name,value)`, `unset(name)`, `reset` (clears all)
Request-scoped: `query`, `select`, `insert`, `create`, `upsert`, `update`, `relate`, `delete` accept `vars` object (key=var name without $), overrides session vars for that call only
Usage in queries: `$var_name`

# Methods
authenticate(token): no desc needed.
  params: [token:str] (required)
  request: {id,method:"authenticate",params:[token_str]}
  response: {id,result:null}

create(thing, data?):
  params: [thing:str(table|record_id) required, data:obj optional]
  request: {method:"create",params:["person",{name:"Mary"}]}
  response: {id,result:[{id, name}]}
  note: passing table generates random ID

delete(thing):
  params: [thing:str(table|record_id) required]
  request: {method:"delete",params:["person:id"]}
  response: {id,result:{ ... deleted record }} (returns deleted record, unlike SQL DELETE)

info():
  no params
  request: {method:"info"}
  response: {id,result:{id, name,...}} (authenticated record user's data)

insert(thing, data):
  params: [thing:str(table) required, data:obj|array required]
  request single: {method:"insert",params:["person",{name:"Mary"}]}
  request bulk: {method:"insert",params:["person",[{...},{...}]]}
  response: {id,result:[{id,name}]}

insert_relation(table, data):
  params: [table:str|null required, data:{in,out,...} required]
  if table null, infers from data.id
  request: {method:"insert_relation",params:["likes",{in:"user:a",out:"post:b"}]}
  response: {id,result:{id,in,out,...}}

invalidate():
  no params, invalidates current session auth
  request: {method:"invalidate"}, response: {id,result:null}

let(name, value): (WebSocket only)
  params: [name:str (no $), value:any]
  request: {method:"let",params:["website","https://..."]}
  response: {id,result:null}

live(table, diff?): (WebSocket only)
  params: [table:str required, diff:bool optional (true->JSON patches)]
  request: {method:"live",params:["person"]}
  response: {id,result:"uuid"} (live query UUID)
  live notifications: {result:{action:"CREATE"|"UPDATE"|"DELETE", id:UUID, result:{record}}}

merge(thing, data?):
  params: [thing:str(table|record_id) required, data:obj optional]
  merges data into existing records
  request: {method:"merge",params:["person",{active:true}]}
  response: {id,result:[{id,active,name}]}

patch(thing, patches, diff?):
  params: [thing required, patches:array of JSON Patch ops, diff:bool optional]
  request: {method:"patch",params:["person",[{op:"replace",path:"/x",value:"y"}]]}
  response: {id,result:array of diffs}

ping():
  request: {method:"ping"}, response: {id,result:null}

query(sql, vars?):
  params: [sql:str required, vars:obj optional]
  request: {method:"query",params:["SELECT ...",{tb:"person"}]}
  response: {id,result:[{status:"OK",time:"...",result:[...]}]}

relate(in, relation, out, data?):
  params: [in:str required, relation:str(table) required, out:str required, data:obj optional]
  request: {method:"relate",params:["person:a","knows","person:b"]}
  response: {id,result:{id,in,out}}

reset():
  clears auth, NS/DB, connection vars, aborts live queries
  request: {method:"reset"}, response: {id,result:null}

run(func_name, version?, args?):
  params: [func_name:str (prefix fn:: or ml::), version:str|null optional, args:array optional]
  for ml:: models, version required
  example: {method:"run",params:["ml::classifier","v2.1",["img_data"]]}
  response: {id,result:value}

select(thing):
  params: [thing:str(table|record_id) required]
  request: {method:"select",params:["person"]}
  response: {id,result:array of records}

signin(...):
  params: object containing:
    ROOT/NS/DB login: {user, pass}
    Record login: {NS, DB, AC, ...vars} (AC=access method)
  response: {id,result:token_string} (or null for root?)
  example root: {method:"signin",params:[{user:"root",pass:"..."}]}
  example record: {method:"signin",params:[{NS:"ns",DB:"db",AC:"acc",username:"...",password:"..."}]}

signup(...):
  params: object {NS, DB, AC, ...vars} (record access signup)
  request: {method:"signup",params:[{NS:"ns",DB:"db",AC:"acc",username:"...",password:"..."}]}
  response: {id,result:token_string}

unset(name): (WebSocket only)
  params: [name:str (no $)]
  request: {method:"unset",params:["website"]}, response: {id,result:null}

update(thing, data?):
  params: [thing:str required, data:obj optional]
  replaces record if exists, else empty array
  request: {method:"update",params:["person:id",{name:"John"}]}
  response: {id,result:{id,name}}

upsert(thing, data?):
  params: [thing required, data optional]
  replaces or creates record
  request: similar to update
  response: {id,result:{record}}

use(ns, db):
  params: [ns:str|null|none, db:str|null|none]
  null unsets, none no change, string sets
  request: {method:"use",params:["ns","db"]}, response: {id,result:null}
  Cannot unset only DB (must unset both or ns)

version():
  no params
  response: {id,result:{version, build, timestamp}}

# HTTP REST Endpoints

Base URL: `http://localhost:8000`

Common Headers (often required):
`Authorization`: basic/token (optional if unauthenticated)
`Surreal-NS`: namespace
`Surreal-DB`: database
`Accept`: application/json

# Endpoints

1. GET /status
   returns 200 OK, no body. Checks web server running.

2. GET /health
   returns 200 if server & storage OK, 500 otherwise.

3. GET /version
   returns plain text version string (e.g., `surrealdb-3.0.0-beta`).

4. POST /import
   headers: Authorization, Surreal-NS, Surreal-DB, Accept
   body: SurrealQL file (binary)
   Import data into NS/DB.

5. POST /export
   headers: Authorization, Surreal-NS, Surreal-DB, Accept
   body (optional JSON): options object {users, accesses, params, functions, analyzers, tables, versions, records} (all optional bool/array)
   Exports all data for NS/DB; writes file if `-o` used.

6. POST /signin
   headers: Accept
   body JSON: {ns?, db?, ac?, user?, pass?, ...} (ns required for DB/record, db required for record, ac required for record)
   response JSON: {code, details, token}
   Example record: {"ns":"main","db":"main","ac":"users","user":"johndoe","pass":"123456"}

7. POST /signup
   headers: Accept
   body JSON: {ns, db, ac (access method), ...vars}
   response JSON: {code, details, token}

   Note: must define record access method first (via /sql with DEFINE ACCESS ...)

8. GET /key/:table
   headers: Authorization, Surreal-NS, Surreal-DB, Accept
   translates to: SELECT * FROM type::table($table)
   example: curl -X GET ... http://localhost:8000/key/person

9. POST /key/:table
   headers: Authorization, Surreal-NS, Surreal-DB, Accept
   body: JSON object
   translates: CREATE type::table($table) CONTENT $body
   response: [{result:[{id, ...}], status:"OK", time:"..."}]

10. PUT /key/:table
    headers: same
    body: JSON object
    translates: UPDATE type::table($table) CONTENT $body
    response: similar update all records

11. PATCH /key/:table
    headers: same
    body: JSON object
    translates: UPDATE type::table($table) MERGE $body
    response: updated records

12. DELETE /key/:table
    headers: same
    translates: DELETE FROM type::table($table) RETURN BEFORE
    returns deleted records

13. GET /key/:table/:id
    translates: SELECT * FROM type::record($table, $id)

14. POST /key/:table/:id
    body: JSON object
    translates: CREATE type::record($table, $id) CONTENT $body

15. PUT /key/:table/:id
    body: JSON object
    translates: UPDATE type::record($table, $id) CONTENT $body

16. PATCH /key/:table/:id
    body: JSON object
    translates: UPDATE type::record($table, $id) MERGE $body

17. DELETE /key/:table/:id
    translates: DELETE FROM type::record($table, $id) RETURN BEFORE

18. POST /sql
    headers: Authorization, Surreal-NS, Surreal-DB, Accept
    body: SurrealQL statement(s)
    query params can be provided as URL query string (e.g., ?age=18), replacing $ variables in query (prepared statements)
    response: array of {time, status:"OK", result:[...]}

19. POST /graphql (since v2.0.0)
    headers: Authorization, Surreal-NS, Surreal-DB, Accept
    body: JSON {query: "graphql query string"}
    requires DEFINE CONFIG GRAPHQL AUTO first

20. POST /ml/import
    headers: Authorization, Surreal-NS, Surreal-DB
    body: .surml file (chunked)
    imports ML model

21. GET /ml/export/:name/:version
    headers: Authorization, Surreal-NS, Surreal-DB, Accept
    exports model to .surml file

22. Custom API: /api/:namespace/:database/:endpoint
    defined via DEFINE API statement; supports GET, POST, etc.
    headers: Authorization, Surreal-NS, Surreal-DB
    example: DEFINE API "/custom" FOR GET MIDDLEWARE ... THEN { { status:200, body:{...} } }
    request: curl http://localhost:8000/api/ns/db/custom -H "Surreal-NS: ns" -H "Surreal-DB: db"

# Authentication Notes
Token can be used via Authorization header (e.g., Bearer token) or via signin/signup response.
For HTTP, set Surreal-NS/Surreal-DB headers to specify namespace/database.
`use` method in RPC to set context.
Record access method: signup/signin with NS, DB, AC fields.

All methods return JSON: {id, result} or {id, error:{code,message}}.

Custom APIs (Middleware): API handlers defined with `DEFINE API "/path" ... THEN { response }`. Middleware list runs before handler; built-in helpers like `api::timeout` plus custom `fn::` functions. Custom middleware receives `$req` (object) and `$next` (function), plus optional extra arguments. Must return response object (merged). Example: `fn::increment_num($req, $next) { LET $res = $next($req); $res + { body: { num: $res.body.num + 1 } } }`. Chain order: outer first. Use `api::invoke("/path", {body})` for testing.

Managing APIs: Custom API routes live under `/api/:ns/:db/...`. Inside handler, `$request` provides method, body, headers, query, params (from path captures), context. Path patterns: `:param` captures segment, `*param` captures rest of URL. Example: `DEFINE API "/test/:id"` matches single segment; `"/test/*id"` matches multi-segment. Built-in helpers like `api::timeout` in MIDDLEWARE.

Performance Best Practices: SurrealDB architecture is document-on-key-value, not columnar; compute separated from storage. SurrealDB Cloud for scaling. Server: set log level to info (or warn/error) for production, not debug/trace/full; use `rocksdb` storage engine (e.g., `surreal start --log info rocksdb://path`). Embedded Rust: add `[profile.release] codegen-units=1, lto=true, opt-level=3`, enable `allocator` feature; use tokio multi-thread runtime with `thread_stack_size(10*1024*1024)`. Tauri: disable logger plugin (`"enabled": false` in tauri.conf.json) or build with `TAURI_LOG_LEVEL=off`. Queries: direct record ID selection (`SELECT * FROM user:ID`) avoids table scan; use comma-sep or range IDs (`user:1..=10`). Boolean fields in WHERE improve speed. Indexes: `DEFINE INDEX ...`; use `EXPLAIN` to see index usage; `UPDATE/DELETE` with `WHERE` may not use index (pre-2.3?); workaround: subquery `UPDATE (SELECT id FROM table WHERE ...) SET ...`. `UPSERT` with unique index avoids scan: `UPSERT table SET email = ..., name = ...` uses index to locate. Index lookup on remote fields: define index on `user.role` and `access.user`, then `SELECT * FROM access WHERE user.role = 'admin'` will use index-based join.

Real-Time & Event-Driven:
- Temporal querying with complex record IDs: use `CREATE weather:["London", time::now()]`. Query range: `weather:["London", start]..=["London", end]` performs partition scan (no separate index needed). Compared to field-based, table scan with index works but range on ID is ~100x faster for large data.
- ULIDs/UUIDs: built-in `rand::ulid()` and `rand::uuid()` contain creation time in milliseconds; can convert back via `time::from_ulid/$from_uuid` (precision floor to ms).
- Live queries: `LIVE SELECT * FROM table WHERE ...` returns UUID subscription; results pushed on create/update/delete. Optionally use `DIFF` for JSON Patch diff on update. Listen via SDKs (e.g., Rust `db.select("table").live()`).
- Events: `DEFINE EVENT name ON table WHEN condition THEN { ... }`. Can use `http::post` etc.; avoid recursive event loops (max computation depth). For aggregations, prefer table views.
- Table views: `DEFINE TABLE sum AS SELECT ... FROM purchases GROUP BY ...` creates virtual table with predictable complex IDs based on group fields. Query directly by ID (instant). Use `DROP` on source if raw table not needed.

Live Queries: `LIVE SELECT ...` returns UUID immediately; notifications stream: `{action: "CREATE"|"UPDATE"|"DELETE", result: record}`. `DIFF` sends JSON Patch arrays. Filter with `WHERE`. Consistency: reflects committed data; ordering not total across writers. Parameters: since v3, session vars allowed (`LET $x = ...; LIVE SELECT * FROM type::table($x)`). Kill with `KILL uuid`.

Changefeeds: Define retention with `DEFINE TABLE name CHANGEFEED 3d` (or database-level). Replay using `SHOW CHANGES FOR TABLE name SINCE datetime` or `SINCE versionstamp` with limit. Returns array of `{changes: [...], versionstamp: u64}`. Each change can be `define_table` or `update` (full record). With `INCLUDE ORIGINAL`, updates carry reverse diff. Versionstamp monotonically increments; use as cursor for pagination.

HTTP via /sql: POST to `<host>/sql` with headers: `Authorization` (optional), `Accept: application/json`, `surreal-ns`, `surreal-db`. Body raw SurrealQL string. Response: array of query results with `time`, `status`, `result`. Example: `curl -X POST -u root:secret -H "surreal-ns: main" -H "surreal-db: main" -H "Accept: application/json" -d "SELECT * FROM person" http://localhost:8000/sql`.

SDKs: Supported: Rust, JavaScript/TypeScript, Python, Node.js, .NET, Golang, Java, PHP. Each provides `query(sql, vars?)` method. JavaScript: `db.query<T>(query, vars)` returns array of results; `.query_raw()` for raw RPC response. PHP: `db->query($query, $vars)`. Python: `db.query(sql, vars)`. .NET: `db.Query(sql)`, params passed within SQL string (automatically parameterized); also `db.RawQuery(sql, params)` for explicit parameters. Golang: `db.Query(sql, vars)`. Rust: `db.query(sql).bind(vars)`, results can be taken by index and deserialized. Vars are bound as named parameters (e.g., `$tb`). All enforce prepared statements, preventing injection.

SurrealDB Security (compressed)

Authentication: System users (root/ns/db level, RBAC with roles OWNER/EDITOR/VIEWER via DEFINE USER) and Record users (defined via DEFINE ACCESS TYPE RECORD with custom SIGNUP/SIGNIN queries in SurrealQL, no RBAC, permissions enforced per table/field). Sessions established after auth; $session and $token vars hold identity/claims. Tokens are JWTs; duration configurable with DURATION FOR SESSION/TOKEN. System user passwords stored hashed with Argon2id. For record users, use crypto::argon2::generate/compare.

Permissions (Row-Level Security): Table permissions: PERMISSIONS FOR select/create/update/delete WHERE expr. Field permissions: FOR select/create/update. Expressions can reference $auth (authenticated record user’s fields) or $session. All denied by default (NONE). Example: table user PERMISSIONS FOR select WHERE id = $auth.id; field internal_note PERMISSIONS FOR update WHERE $auth.role = 'admin'. Guest access (--allow-guests) still obeys permissions.

Capabilities: Disabled by default. Controlled via --allow-* / --deny-* flags (also env vars). Priority: specific over general, deny over allow. Examples: --deny-all --allow-funcs "array,string,crypto::argon2,http::get" --allow-net "api.example.com:443". Functions: allow/deny lists; network targets: IPs, hosts, CIDR; scripting; experimental (surrealism,files); arbitrary queries per user group (guest,record,system). Guest access: allows unauthenticated queries but only if resource PERMISSIONS FULL. Network DNS resolved, but no reverse DNS; recommend allow by default with specific --allow-net, never --allow-net --deny-net (denylist approach risky). Deny arbitrary query can block /sql, /key, RPC methods for given groups; use DEFINE API for controlled access.

Tokens & JWTs: SurrealDB verifies JWTs. DEFINE ACCESS ... TYPE JWT (system) or TYPE RECORD WITH JWT. Supports symmetric (HS256) with secret, asymmetric (RS256, ES256) with public key or JWKS URL. Claims mapped to $auth and $token. Prefer asymmetric + JWKS for production. Record access can also issue JWTs after custom signin. Token and session expiration are separate: tokens stored client-side (short-lived), sessions server-side (can be longer). Best to use token only to establish session then discard.

Sessions: $session contains ip, exp, ns, db, rd (record id), ac (access method), or (origin), id (client id), tk (token claims). $token contains token claims (iat, nbf, exp, jti, iss, NS, DB, AC, ID). $auth is the record of authenticated record user (only for record users). Session/connection remains until expired or invalidated; use separate WebSocket connections per user for isolation.

Best practices (condensed):
- Capabilities: deny all by default, allow only necessary specific capabilities. Use allowlisting, not denylisting.
- Passwords: never store plaintext; use crypto::argon2::generate (or bcrypt/scrypt/pbkdf2). Never use fast hashes (md5, sha1). Store password hashes with field/table permissions to prevent read.
- Expiration: always set DURATION FOR SESSION and TOKEN. Session short (hours), token extremely short (seconds).
- Query safety: always bind parameters (prepared statements) via vars/bind, never concatenate user input.
- Content safety: use string::html::encode to prevent XSS; string::html::sanitize if HTML required but less safe.
- JWTs: use asymmetric algorithms (RS/ES) with JWKS; avoid symmetric secrets in production.
- Network exposure: limit to internal networks; if public, use --no-identification-headers to hide version, and use firewall/IPS.
- Least privilege: define system users at lowest level with minimal role (VIEWER/EDITOR).
- Encryption in transit: use TLS (--web-crt/--web-key) or reverse proxy.
- Encryption at rest: use disk encryption (LUKS/BitLocker); datastore encryption optional.
- Isolate WebSocket sessions per user; avoid reusing connections across users.
- Token storage: avoid storage if possible; if needed, use browser localStorage with CSP and XSS prevention; never cookies (CSRF risk).

# RELATE
Creates graph edges between records. Edge tables auto-delete when no relations remain. Required fields: `in`, `out` (direction-specifying, type-restrictable but not removable).

Syntax: `RELATE [ONLY] @from -> @table -> @to [CONTENT @val | SET @f=@v ...] [RETURN ...] [TIMEOUT @dur]`

Note: RELATE works on non-existent records (use `ENFORCED` on table to prevent this).

Examples:
```surql
RELATE person:a->wrote->article:b;
RELATE [cat:a, cat:b]->parent_of->cat:c;  -- creates 2 edges
RELATE [a,b]->[c,d];                       -- creates 4 edges (product)
RELATE p:a->wrote->a:b SET meta.loc="Tallinn";
RELATE ONLY p:a->wrote->a:b;              -- returns object not array

Delete edge: DELETE person:a->bought WHERE out=product:b RETURN BEFORE Enforce
direction: DEFINE FIELD in ON wrote TYPE record<person>; DEFINE FIELD out ON
wrote TYPE record<article>; Unique relations: DEFINE INDEX uniq ON wrote COLUMNS
in,out UNIQUE;

Graph Queries

Arrows: -> (out), <- (in), <-> (both directions)

SELECT ->wrote->article FROM person;           -- articles written by persons
SELECT <-wrote<-person FROM article;           -- persons who wrote articles  
SELECT ->wrote.time_written AS t FROM person;  -- edge field
SELECT id, ->wrote->(book WHERE "Ring" IN name).* FROM person;  -- filter at node
SELECT ->(wrote WHERE written_at="Athens")->book.* FROM person; -- filter at edge
SELECT array::complement(<->sister_of<->city,[id]) AS sisters FROM city; -- bidirectional dedup

Wildcards: ->? (any table), ->(?).* AS out_relations (all with data) Aliases:
SELECT ->f->c->f->c AS friends_of_friends FROM cat:one Parentheses for clauses:
->(SELECT *, time::now() AS t FROM knows LIMIT 1) Range on edges:
person:one->likes:2..=4->person

Recursive: SELECT @.{3}->to->city FROM city:1 (3 levels),
city:1.{1..20}->to->city (range), @.{..} (open, max 256)

SELECT @.{1..5}.{ id, next: ->to->city } FROM city;  -- range + destructure

Use TIMEOUT with recursive queries.

Graph clauses (v2.2+): Full SELECT syntax inside graph path:

SELECT ->(SELECT count() AS c, know_in_person FROM likes GROUP BY know_in_person) FROM person;

LIVE SELECT

Real-time change stream. Returns UUID (live query ID). Use to KILL.

Syntax: LIVE SELECT [VALUE @fields | DIFF] FROM @targets [WHERE @cond] [FETCH
@fields]

LIVE SELECT * FROM person;
LIVE SELECT DIFF FROM person;              -- JSON Patch format
LIVE SELECT * FROM person WHERE age > 18;
LIVE SELECT * FROM person WHERE age > 18 FETCH friends;  -- v2.2+

  - Only committed transactions publish notifications
  - Single-node only (multi-node WIP)
  - Parameters in WHERE supported v3.0+: LIVE SELECT * FROM type::table($t)
    WHERE loc=$loc

LET

Store values/query results as parameters ($ prefix).

Syntax: LET $param [: @type] = @value

[CHANGEFEED @dur [INCLUDE ORIGINAL]] [PERMISSIONS ...] [COMMENT @s]

SURREALDB LANGUAGE PRIMITIVES - COMPRESSED REFERENCE

=== IDIOMS (PATH NAVIGATION) ===

Parts: field (.name), index ([0]), all (.*), last ([$]), where filter ([WHERE]), method (.method()), graph (->/->/etc), destructure (.{a,b}), optional (.?), recursive (.{n})

Field: SELECT address.city FROM person
Index: SELECT results[0].score FROM student
All: SELECT results.* FROM student  -- returns full array
Last: SELECT results[$].score FROM student
Filter: SELECT friends[WHERE age > 18].name FROM person:5
Method: SELECT name.uppercase() FROM person  -- maps to string::uppercase()
Optional (.?): SELECT person.spouse.?.name FROM person  -- returns NONE if spouse missing

Destructure:
  SELECT obj.{ a, c.{ e, f } } FROM ONLY person:1
  SELECT * OMIT obj.c.{ d, f } FROM ONLY person:1
  Alias inside destructure: $town.{ location, num_people: population }
  AS is for SELECT aliases; colon(:) is for destructure aliases

Current record (@, v2.1+):
  SELECT @.{ name, id, orbits: ->orbits->star } FROM planet
  SELECT @.{ name, id, orbits: ->orbits->star }.values() FROM planet
  @ is required in recursive SELECT queries

Expressions in destructure (v3+):
  person:one.{ name, name_length: name.len(), accessed_at: time::now() }
  $this = current object, $parent = parent object in nested destructure

=== RECURSIVE PATHS (v2.1+) ===

Syntax: record.{depth}(path_to_repeat).rest
  planet:earth.{2}->has->(?)          -- exact 2 hops
  planet:earth.{1..4}->has->(?)       -- 1 to 4 hops
  planet:earth.{..}->has->(?)         -- open-ended (max 256)
  planet:earth.{..}->has->(?).name    -- WRONG: recurses 'path.name'
  planet:earth.{..}(->has->(?)).name  -- RIGHT: recurses path, then .name

Depth range: 1..256. {0} and {500} error. Open-ended {..} stops at dead end.
Dead ends: NONE, NULL, empty array, or same value as previous iteration.
Below minimum depth -> returns NONE. Above minimum -> returns last valid.

Using @ for tree building:
  planet:earth.{1..2}.{ name, id, contains: ->has->(?).@ }
  .@ marks the recursive gateway field. Only one .@ allowed per query.

In SELECT: SELECT @.{1..4}(->knows->person).name AS names FROM person

Works with record links too:
  person:tobie.{..}(.friends).name

=== PATH/COLLECT/SHORTEST ALGORITHMS (v2.2+) ===

  person:you.{..+path}->knows->person        -- all possible paths (array of arrays)
  person:you.{..+collect}->knows->person     -- unique nodes BFS order
  person:you.{..+shortest=person:star}->knows->person  -- shortest path
  Add +inclusive to include starting record
  Bounded: person:you.{..2+collect}->knows->person
  Chain: person:you.{..+shortest=person:star+inclusive}(->knows->person).name.map(|$n| n.uppercase())
  DO NOT combine these algorithms with .@ (causes error)

=== RECORD LINKS ===

Store record IDs in fields: CREATE person:jaime SET friends = [person:tobie, person:simon]
Traverse: SELECT friends.name FROM person:tobie  -- fetches remote records async
Deep: SELECT friends.friends.friends.name FROM person:tobie  -- unlimited depth, nested arrays
Direct access: person:tobie.*  /  person:tobie.{ name, email }

=== RECORD REFERENCES (v2.2+) ===

Add REFERENCE to DEFINE FIELD for record/array<record> fields (top-level only).
Query incoming refs with <~ (like graph but reverse):
  SELECT *, <~person AS owners FROM comic_book
  SELECT *, <~person.{ id, name } AS owners FROM comic_book

Computed field for refs:
  DEFINE FIELD owned_by ON comic_book COMPUTED <~person
  DEFINE FIELD owned_by ON comic_book COMPUTED <~(person FIELD comics)  -- narrow by field
  DEFINE FIELD all_readers ON comic_book COMPUTED <~(person FIELD comics borrowed_comics)

ON DELETE behaviours (set in REFERENCE clause):
  ON DELETE IGNORE    -- default, stale refs remain
  ON DELETE UNSET     -- removes deleted ref from field
  ON DELETE CASCADE   -- deletes the referencing record
  ON DELETE REJECT    -- prevents deletion if referenced (error)
  ON DELETE THEN { UPDATE $this SET ...; }  -- custom logic; $this=record, $reference=deleted ref
  Note: UNSET on non-optional field = effectively REJECT

  Numbers outside int64 range stored as strings with backticks

  Imprecise float cast: <decimal>8.888888888888888 = 8.88888888888889dec (imprecise)
  Precise suffix: 8.888888888888888dec = 8.888888888888888dec (exact)

This is the classic Document (Nested/OOP) vs. Relational (Referential/Normalized) database design dilemma.
Because you are using SurrealDB (a multi-model database) and your primary goals are "reducing headaches," "implicitly enforced integrity," and "low complexity," my strong recommendation is to stick to Nested Objects (array<object>) for Document Lines (Orders, Bills, Invoices), while keeping Ledgers (Payments, Adjustments, Inventory Exchanges) as Referential tables.
Here is the deep-dive architectural breakdown of why, how they compare, and what the industry standard dictates.
1. The Industry Standard (ERP & Accounting)
In legacy SQL databases (QuickBooks, old SAP), lines must be a separate table (order_header and order_line) because SQL cannot natively store arrays.
However, in modern Document databases (and multi-model DBs like SurrealDB), the industry standard for commerce/invoicing is Nested/Embedded arrays, unless the array is unbounded.
Bounded Data: An Order or Invoice typically has 1 to 100 lines. It rarely grows forever. This belongs inside the document.
Unbounded Data: A customer's Payment History or an Inventory item's movement ledger grows infinitely over time. This belongs in separate tables (which you already correctly modeled as payment and exchange).
2. Complexity & Maintenance (The "Headache" Factor)
Approach A: Nested Objects (Current Design) - Winner for Low Complexity
If the frontend sends an updated Order with an edited line, a deleted line, and a newly added line, the backend code is literally one line:
code
Surrealql
-- The DB automatically replaces the array, recomputes the math, checks the assertions, and saves.
UPDATE order:123 SET lines = $frontend_payload;
Why it reduces headaches:
Zero Diffing Logic: You don't have to write code to figure out which lines the user deleted, which they added, and which they modified.
Atomic Integrity: The entire order (with all its math) is evaluated in a single split-second transaction. If the math fails, the whole document rejects instantly.
Approach B: Referential Lines (Separate order_line table)
If you move lines to a separate table, your frontend sends the same payload, but your backend now has a massive headache:
You must query existing lines.
You must compare them against the incoming payload.
You must run DELETE on missing lines, UPDATE on changed lines, and CREATE on new lines.
Each of these triggers the Empty Update Pattern event to snapshot the parent order table.
3. Scalability & Performance
Read Performance: Nested is O(1). You ask for order:123 and get the whole invoice perfectly formatted for the UI. Referential requires a query SELECT *, (SELECT * FROM order_line WHERE order = $parent.id) AS lines FROM order:123.
Write Performance (Concurrency): This is the only area where Referential wins. If you have 5 warehouse workers simultaneously modifying different lines of the exact same order at the exact same millisecond, nested objects will throw a lock conflict (Optimistic Concurrency failure). Referential tables avoid this because workers are locking different order_line records. (However, in standard SaaS accounting, multiple users rarely edit the exact same draft invoice simultaneously).
4. How Referential would look in SurrealDB (If you chose to scale it)
If you ever build an EDI system where an order has 50,000 lines, you must use Referential. As you noted, the VALUE clause on the child line cannot update the parent directly. You use the Empty Update Pattern from the SurrealDB 3.0 protocol.
Here is how beautifully it handles it without explicit backend code:
code

For a SaaS Accounting / Commerce module (like Zoho Books or QuickBooks):
Keep order.lines and procurement.lines as Nested Arrays.
It guarantees absolute mathematical integrity in a single transaction.
It cuts your API handler code down to practically zero (just pass the JSON to SurrealDB).
It makes UI state management incredibly easy.
Use Shadow References (which you already have).
By using z_item_refs ON order TYPE set<record<inventory>> ... REFERENCE ON DELETE REJECT, you get the best of both worlds: Nested data structure + Relational Database Protection. A user cannot delete an inventory item if it exists inside an array inside an order.
Keep Ledgers (payment, exchange, adjustment_note) as Referential.
These represent immutable financial events that grow over time. The "Empty Update Pattern" perfectly aggregates these up to the organization_account and order tables.

Yes, strictly. This is called the Order-to-Cash (O2C) lineage.
However, an Invoice Line does not just "look" at the Order Line; it copies the data and holds a reference pointer.
Why? Because if a salesperson updates the price on a Sales Order two months later, the historical, legally-binding Invoice must not change. The Rule of Multiplicity:
1 Order can result in 5 Invoices (Partial Billing).
1 Invoice can be refunded by 3 Credit Notes (Partial Refunds).
1 Credit Note can refund lines from 3 different Invoices.
This is why Normalized Lines (separate tables) are mandatory for a real ERP. You cannot do this with nested arrays.
Part 2: Reactive "Touch" vs Explicit Event Updates (The Performance Trap)
You asked if you should explicitly set the fields in the Event, rather than just "touching" the parent to trigger the VALUE clause.

Line 1 inserts -> Touches Order -> Order runs math::sum() on 1 line.
Line 2 inserts -> Touches Order -> Order runs math::sum() on 2 lines.
...
Line 1000 inserts -> Touches Order -> Order runs math::sum() on 1000 lines.
You just made the database perform 500,000 summation operations for a single 1000-line invoice. At scale, this crashes the CPU.
The Expert Solution:
Keep the "Empty Update" pattern, but you must enforce batch writes at the application layer, or use SurrealDB 3.0's dynamic COMPUTED fields if the document is write-heavy and read-light.
Since invoices are read 100x more than they are written, Stored VALUE clauses with Empty Updates are correct, but your backend must insert lines via batch: INSERT INTO invoice_line [...] so the event triggers efficiently.
