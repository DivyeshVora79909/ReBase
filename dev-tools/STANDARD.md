# 🌌 ReBase ERP: Architecture & Physics Standard (v21)

## The Core Philosophy

ReBase is a mathematically unbreakable, real-time ERP engine built natively on **SurrealDB**.

We do not write thousands of lines of asynchronous API endpoints, CRON jobs, or imperative `IF/THEN` loops to calculate inventory or balances. Instead, we treat the database as a **Deterministic Physics Engine**.

Data flows like electricity through a circuit board. You input the raw facts (an invoice, a stock movement), and the database instantly, synchronously, and securely updates the entire financial reality of the company in $O(1)$ time.

---

## 1. The Alphabetical Lexicon (How Fields Execute)

SurrealDB executes schema fields alphabetically. To ensure math happens in the exact right order—without unpredictable race conditions—we use a strict, 4-tier naming prefix. **This is the only thing a developer needs to memorize.**

| Prefix        | Name                | Purpose                                                                             | Mutation          |
| :------------ | :------------------ | :---------------------------------------------------------------------------------- | :---------------- |
| **`a_`**      | **Inputs**          | Raw data provided by the User/API, and Foreign Keys (e.g. `a_qty`, `a_invoice`).    | Writable by APIs. |
| **`c_`**      | **Computed BI**     | Auto-generated by the compiler. Instant analytics payloads for the frontend.        | Read-Only.        |
| **`d_`**      | **Derivations**     | Mathematical steps. Infinite depth (`d1_`, `d2_`).                                  | Read-Only.        |
| **`e_guard`** | **Physics Shields** | A single block asserting business rules (e.g. `qty > 0`). If it fails, it `THROW`s. | Internal only.    |

**Example of the Lexicon in Action:**

```surrealql
DEFINE FIELD a_invoice ON invoice_line TYPE record<invoice> REFERENCE;
DEFINE FIELD a_qty ON invoice_line TYPE decimal ASSERT $value > 0dec;
DEFINE FIELD a_price ON invoice_line TYPE decimal;

-- Math happens next
DEFINE FIELD d1_vector ON invoice_line VALUE $this.a_invoice.d1_vector;
DEFINE FIELD d2_raw_base ON invoice_line VALUE $this.a_qty * $this.a_price;

-- Physics evaluated last
DEFINE FIELD e_guard ON invoice_line TYPE bool VALUE {
    IF $this.d2_raw_base < 0dec { THROW "MATH_ERR: Negative value not allowed."; };
    RETURN true;
} ASSERT $value = true;
```

_Notice how clean this is. No UI bloat, no temporary context variables. Just raw business physics._

## 2. The Smart Meta-Compiler (The "Automated Plumber")

You **never** manually write database triggers, Row-Level Security, or cascading updates. The Developer writes 3 core logic files; the Compiler writes the remaining 7 files automatically.

The Compiler acts like a stateless robot. It reads your definitions, traces the graph connections, and lays down the internal plumbing.

### What the Compiler Does:

1. **Builds Security & Audit (Files 02, 03, 04, 05):**
   It finds all `a_` fields that are record links. It uses them to build the DAG (Directed Acyclic Graph) Ownership tree. If an Invoice belongs to an Org, the Org's owners automatically inherit RLS permissions to the Invoice.
2. **Builds the B-Tree Indexes (File 11):**
   It automatically generates indexes for all your root foreign keys and BI View groupings to ensure operations remain lightning-fast.
3. **Builds the "Elevator Going Up" (File 08 & 12):**
   When you write an analytical View (e.g., grouping Invoices by Org), the compiler detects it. It creates an Upward Ping (File 08) so the Org knows when a child updates. Better yet, it auto-generates a `COMPUTED` field (File 12) on the Org itself. When the frontend queries the Org, the database automatically attaches the view analytics inside the payload!
4. **Builds the "Elevator Going Down" (File 09):**
   If a child table relies on a grand-parent's field (e.g., `$this.a_from.a_invoice.a_currency`), the compiler spots this dependency in your code. It automatically writes an event on the `org` table that says: _"If the currency changes, traverse the native graph (`<~invoice<~sl`) and alert the child to recalculate."_

## 3. The Two Types of Views

In traditional databases, views are just saved queries. In ReBase, Materialized Views are in-memory analytical cubes. We separate them strictly into two categories:

### A. The BI Cubes (`07_views.surql`)

These are your analytics. They use `GROUP BY` to roll data up to parents.

- _Rule:_ Pack them with statistics! (`math::mean`, `math::max`, `math::variance`).
- _Compiler Action:_ The compiler scans this file and automatically attaches these rich insights to the Parent tables as `c_` fields.

### B. The Physics Matrix (`10_config.surql`)

Some views exist _only_ to enforce reality. For example, ensuring a Warehouse doesn't ship more of an Item than it physically possesses requires grouping by `[warehouse, item]`.

- _Rule:_ We place these complex, multi-dimensional security views in `10_config.surql`.
- _Compiler Action:_ Because they aren't in File 07, the compiler safely ignores them, preventing the UI from being bloated with massive matrices of data.

## 4. State Machines over Black Boxes

Accounting systems run on semantics, not just math.
Instead of multiplying amounts by hidden vectors (`+1` or `-1`) to figure out if money is moving in or out, we rely on **Explicit State Machines**.

- _Payments:_ `inbound`, `outbound`, `internal`
- _Invoices:_ `purchase`, `sales`

This makes the database self-documenting. If a developer looks at a record, they know exactly what it is. The Views handle the heavy lifting: `math::sum(IF state = 'sales' THEN amount ELSE 0 END)`.

## 5. The Developer Workflow

When building features in ReBase, you only touch three places:

1.  **`06_table_fields.surql`**: Define the raw inputs (`a_`), the math (`d_`), and the physics constraints (`e_guard`).
2.  **`07_views.surql`**: Define the Business Intelligence roll-ups using `GROUP BY`.
3.  **`10_config.surql`**: Define extreme edge-case validations (like warehouse stock limits).

Once written, you run `node compiler.js`. The compiler reads your logic, maps the graph, and flawlessly generates the millions of potential cascades, security policies, and real-time BI fields required for a global ERP.

### Frontend Guidelines

- **To Read Data:** Simply fetch the record. The auto-generated `c_` fields will instantly provide all historical and analytical context.
- **To Write Data:** Only ever mutate `a_` (input) fields. The database will handle the rest in milliseconds.
