Custom APIs (Middleware): API handlers defined with `DEFINE API "/path" ... THEN { response }`. Middleware list runs before handler; built-in helpers like `api::timeout` plus custom `fn::` functions. Custom middleware receives `$req` (object) and `$next` (function), plus optional extra arguments. Must return response object (merged). Example: `fn::increment_num($req, $next) { LET $res = $next($req); $res + { body: { num: $res.body.num + 1 } } }`. Chain order: outer first. Use `api::invoke("/path", {body})` for testing.

Managing APIs: Custom API routes live under `/api/:ns/:db/...`. Inside handler, `$request` provides method, body, headers, query, params (from path captures), context. Path patterns: `:param` captures segment, `*param` captures rest of URL. Example: `DEFINE API "/test/:id"` matches single segment; `"/test/*id"` matches multi-segment. Built-in helpers like `api::timeout` in MIDDLEWARE.

Performance Best Practices: SurrealDB architecture is document-on-key-value, not columnar; compute separated from storage. SurrealDB Cloud for scaling. Server: set log level to info (or warn/error) for production, not debug/trace/full; use `rocksdb` storage engine (e.g., `surreal start --log info rocksdb://path`). Embedded Rust: add `[profile.release] codegen-units=1, lto=true, opt-level=3`, enable `allocator` feature; use tokio multi-thread runtime with `thread_stack_size(10*1024*1024)`. Tauri: disable logger plugin (`"enabled": false` in tauri.conf.json) or build with `TAURI_LOG_LEVEL=off`. Queries: direct record ID selection (`SELECT * FROM user:ID`) avoids table scan; use comma-sep or range IDs (`user:1..=10`). Boolean fields in WHERE improve speed. Indexes: `DEFINE INDEX ...`; use `EXPLAIN` to see index usage; `UPDATE/DELETE` with `WHERE` may not use index (pre-2.3?); workaround: subquery `UPDATE (SELECT id FROM table WHERE ...) SET ...`. `UPSERT` with unique index avoids scan: `UPSERT table SET email = ..., name = ...` uses index to locate. Index lookup on remote fields: define index on `user.role` and `access.user`, then `SELECT * FROM access WHERE user.role = 'admin'` will use index-based join.

Real-Time & Event-Driven:
- Temporal querying with complex record IDs: use `CREATE weather:["London", time::now()]`. Query range: `weather:["London", start]..=["London", end]` performs partition scan (no separate index needed). Compared to field-based, table scan with index works but range on ID is ~100x faster for large data.
- ULIDs/UUIDs: built-in `rand::ulid()` and `rand::uuid()` contain creation time in milliseconds; can convert back via `time::from_ulid/$from_uuid` (precision floor to ms).
- Live queries: `LIVE SELECT * FROM table WHERE ...` returns UUID subscription; results pushed on create/update/delete. Optionally use `DIFF` for JSON Patch diff on update. Listen via SDKs (e.g., Rust `db.select("table").live()`).
- Events: `DEFINE EVENT name ON table WHEN condition THEN { ... }`. Can use `http::post` etc.; avoid recursive event loops (max computation depth). For aggregations, prefer table views.
- Table views: `DEFINE TABLE sum AS SELECT ... FROM purchases GROUP BY ...` creates virtual table with predictable complex IDs based on group fields. Query directly by ID (instant). Use `DROP` on source if raw table not needed.

Live Queries: `LIVE SELECT ...` returns UUID immediately; notifications stream: `{action: "CREATE"|"UPDATE"|"DELETE", result: record}`. `DIFF` sends JSON Patch arrays. Filter with `WHERE`. Consistency: reflects committed data; ordering not total across writers. Parameters: since v3, session vars allowed (`LET $x = ...; LIVE SELECT * FROM type::table($x)`). Kill with `KILL uuid`.

Changefeeds: Define retention with `DEFINE TABLE name CHANGEFEED 3d` (or database-level). Replay using `SHOW CHANGES FOR TABLE name SINCE datetime` or `SINCE versionstamp` with limit. Returns array of `{changes: [...], versionstamp: u64}`. Each change can be `define_table` or `update` (full record). With `INCLUDE ORIGINAL`, updates carry reverse diff. Versionstamp monotonically increments; use as cursor for pagination.

HTTP via /sql: POST to `<host>/sql` with headers: `Authorization` (optional), `Accept: application/json`, `surreal-ns`, `surreal-db`. Body raw SurrealQL string. Response: array of query results with `time`, `status`, `result`. Example: `curl -X POST -u root:secret -H "surreal-ns: main" -H "surreal-db: main" -H "Accept: application/json" -d "SELECT * FROM person" http://localhost:8000/sql`.

SDKs: Supported: Rust, JavaScript/TypeScript, Python, Node.js, .NET, Golang, Java, PHP. Each provides `query(sql, vars?)` method. JavaScript: `db.query<T>(query, vars)` returns array of results; `.query_raw()` for raw RPC response. PHP: `db->query($query, $vars)`. Python: `db.query(sql, vars)`. .NET: `db.Query(sql)`, params passed within SQL string (automatically parameterized); also `db.RawQuery(sql, params)` for explicit parameters. Golang: `db.Query(sql, vars)`. Rust: `db.query(sql).bind(vars)`, results can be taken by index and deserialized. Vars are bound as named parameters (e.g., `$tb`). All enforce prepared statements, preventing injection.
